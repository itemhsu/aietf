🏗 系統架構模組

六大核心模組，各司其職、低耦合設計

🔌 src/data/ — 資料層

  • alpaca_client.py — REST API 封裝＋指數退避重試
  • market_data.py — 即時報價、市值排行
  • pe_ratio.py — yfinance 本益比抓取
  • benchmark.py — QQQ / SPY 歷史 NAV
  • demo_data.py — DEMO_MODE 假資料（不需 API Key）

⚙️ src/engine/ — 核心引擎

  • strategy_loader.py — 動態載入策略 JSON
  • account_manager.py — 多帳戶管理
  • selector.py — 選股邏輯（市值/動能）
  • rebalancer.py — 再平衡計算（整數股）
  • trader.py — 下單執行＋DRY_RUN 保護

📊 src/report/ — 報告層

  • model.py — 資料收集 → 存為 JSON（Model）
  • view_email.py — Jinja2 HTML 渲染（View）
  • 歷史報告存 reports/YYYY-MM-DD/
  • benchmark_nav_history 預存入報告
  • 回撤 / NAV / P&L 完整欄位

📈 src/dashboard/ — 儀錶板

  • data.py — 純 JSON 讀取（可獨立測試）
  • app.py — Streamlit UI（零 API 呼叫）
  • NAV 折線圖＋QQQ/SPY 勾選比對
  • 回撤圖、持倉表、Top10、觀察清單
  • 部署 Streamlit Community Cloud（免費）

🔔 src/notify/ — 通知層

  • email_sender.py — Gmail SMTP 發送
  • trade_alert.py — 即時交易警報
  • 每日 06:00 台灣時間寄送
  • Email 無設定時靜默略過

☁️ .github/workflows/ — CI/CD

  • daily_workflow.yml — 每日自動執行主流程
  • 三模式：demo / dry_run / live
  • 失敗自動開 GitHub Issue
  • 報告 JSON 自動 commit 回 repo
  • 執行日誌上傳 Artifacts（保留 14 天）

📐 核心設計原則

這些原則不輕易改動，確保系統長期可維護

📄
策略即 JSON
所有投資策略存於 strategies/*.json，新增策略只需新增 JSON 檔，零 Python 代碼修改。
🔀
報告 Model / View 分離
model.py 只產資料存 JSON；view_email.py 只渲染 HTML。互不耦合，可獨立替換。
🔢
只買整數股
所有買賣量用 math.floor()，永遠不下分數股訂單，避免 API 拒絕。
👥
多帳戶架構
accounts/accounts_index.json 管理所有帳戶。同一帳戶同時只能用一個策略，但可隨時切換。
🛡️
DRY_RUN 優先
預設 DRY_RUN=true，需明確設定 false 才實際下單。最大程度防止誤操作損失。
☁️
全雲端執行
所有程序在 GitHub Actions 執行，不依賴本機環境。任何有網路的地方都能觸發。
⚖️
再平衡觸發條件
① 每月初（月份第一個交易日）② 新資金進入（現金超過 NAV 的閾值百分比）③ 初次建倉。
📦
Dashboard 零 API
Dashboard 只讀 reports/*.json，不呼叫任何外部 API。QQQ/SPY 歷史由 GitHub Actions 預存。

📋 策略 JSON 規格

新增策略只需新增一個 JSON 檔到 strategies/ 目錄，無需修改任何 Python 代碼

// strategies/top10_marketcap.json — 市值前十等權重策略 { "strategy_id": "top10_marketcap_v1", "name": "NASDAQ 市值前 10 等權重", "description": "每日選出 NASDAQ 市值最大的 10 檔，各配置 10%", "universe": [ "AAPL", "MSFT", "NVDA", "GOOGL", "AMZN", "META", "TSLA", "AVGO", "TSM", "ASML" // ... 最多 25 檔候選池 ], "selection": { "method": "market_cap", // market_cap | momentum | custom "top_n": 10 }, "allocation": { "weight_per_stock": 0.10, // 各股 10% "cash_buffer": 0.02, // 保留 2% 現金 "share_type": "integer_only" // 只買整數股 }, "rebalance": { "tolerance_pct": 2.0, // 偏離 ±2% 才再平衡 "monthly_first_day": true, // 每月初強制再平衡 "on_new_capital_pct": 5.0 // 新資金超過 5% NAV 觸發 }, "watchlist_categories": { "ai_tech": ["NVDA", "MSFT", "GOOGL", "META", "AMZN"], "semicon": ["AVGO", "TSM", "ASML", "AMD", "QCOM"], "growth": ["NFLX", "UBER", "SNOW", "PLTR", "NOW"] } }

💡 新增策略步驟：複製上方 JSON → 修改 strategy_id → 在 accounts_index.json 將帳戶的 active_strategy 改為新 ID → 觸發 dry_run 驗證。

🔐 環境變數 / GitHub Secrets

所有敏感資訊存於 GitHub Secrets，代碼中只存環境變數名稱

變數名稱說明是否必要
ALPACA_KEY_A 帳戶 A 的 Alpaca API Key ID dry_run / live 模式必要
ALPACA_SECRET_A 帳戶 A 的 Alpaca API Secret dry_run / live 模式必要
ALPACA_URL_A 帳戶 A 的 API Base URL 預設 paper-api.alpaca.markets
EMAIL_USER 發信 Gmail 帳號 可選（無設定則靜默跳過）
EMAIL_PASS Gmail 應用程式密碼（非登入密碼） 可選
DEMO_MODE true = 使用假資料，不需 API Key 由 workflow 自動設定
DRY_RUN true = 計算但不實際下單 預設 true，需明確設 false
ACCOUNT_FILTER 指定單一帳戶 ID（留空 = 全部帳戶） 可選

🔄 三種執行模式

🎭 demo
使用假資料，無需任何 API Key。適合測試 GitHub Actions 流程是否正常。
🔍 dry_run
使用真實 API 取市場資料，計算再平衡訂單，但不實際下單。
🚀 live
使用真實 API，實際執行買賣訂單。需 Live Trading Key + 充足資金。
